# CADE 系统架构文档

## 概述

CADE (Cognitive Agent for Domestic Environment) 是一个开源的具身AI机器人平台，专为家庭服务机器人设计。该系统集成了先进的语音识别(ASR)、语音合成(TTS)和大语言模型(LLM)，创建了一个能够理解自然语言指令、根据上下文做出决策并在家庭环境中执行适当动作的智能助手。

## 设计理念

CADE 采用模块化架构设计，遵循**关注点分离**原则，将系统划分为以下核心层：

1. **Brain层 (大脑)** - LLM智能推理层
2. **Body层 (躯体)** - 机器人物理执行层  
3. **Bridge层 (桥接)** - ROS集成与通信层
4. **Controller层 (控制器)** - 主协调与业务流程层

## 核心组件详细说明

### 1. Brain层 (大脑)

**位置**: `brain/`

#### 1.1 LLM客户端 (`llm_client.py`)
- **功能**: 封装OpenAI兼容接口，支持云端/本地LLM无缝切换
- **关键特性**:
  - 支持DeepSeek、DashScope等云端API
  - 支持本地Ollama实例 (Qwen3等模型)
  - 自动JSON解析和重试机制
  - 同步(`LLMClient`)和异步(`AsyncLLMClient`)版本
- **配置**: 从`config.py`获取运行模式(CLOUD/LOCAL)、API密钥、模型等参数

#### 1.2 提示词系统 (`prompts.py`)
- **功能**: 定义机器人的行为规范、动作空间和输出格式
- **提示词变体**:
  - `default`: 标准提示词，包含详细思考过程
  - `compact`: 精简提示词，不需要thought字段
  - `simple`: 简化提示词，用于测试
  - `debug`: 调试提示词，输出极其详细的推理过程
- **系统提示词内容**:
  - 定义6种核心动作: navigate, search, pick, place, speak, wait
  - 行为规则: 意图识别、思考模式、输出约束
  - 输出格式: 严格的JSON结构

#### 1.3 数据模型 (`schemas.py`)
- **功能**: 使用Pydantic严格定义机器人的动作空间和决策输出格式
- **核心模型**:
  - `NavigateAction`: 导航动作，移动到指定位置
  - `SearchAction`: 搜索动作，寻找物体
  - `PickAction`: 抓取动作，拾取物体
  - `PlaceAction`: 放置动作，将物体放到指定位置
  - `SpeakAction`: 说话动作，语音输出
  - `WaitAction`: 等待动作，保持当前状态
  - `RobotAction`: 所有动作的联合类型
  - `RobotDecision`: LLM决策输出格式，包含thought、reply、action字段

### 2. Body层 (躯体)

**位置**: `body/`

#### 2.1 机器人接口 (`robot_interface.py`)
- **功能**: 定义统一的机器人抽象接口
- **核心内容**:
  - `RobotState`枚举: IDLE, THINKING, EXECUTING, SPEAKING, ERROR
  - `RobotInterface`抽象类: 定义所有机器人必须实现的6个方法
  - 状态管理方法: `get_state()`, `set_state()`

#### 2.2 机器人实现 (`robot.py`)
- **功能**: CADE机器人V2实现，继承V1的空间认知与V2的语音交互
- **关键特性**:
  - V1基础: 内部语义地图、物体数据库、动作状态校验
  - V2增强: ROS语音交互集成、状态机对齐、回声抑制
  - 语义地图: 包含home, kitchen, living_room, bedroom等位置
  - 物体数据库: apple, bottle, cup, book等物体信息
  - 回声抑制: `is_busy()`方法检查机器人是否忙碌
- **动作实现**: navigate, search, pick, place, speak, wait

### 3. Controller层 (控制器)

**位置**: `robot_controller.py`

#### 3.1 机器人控制器 (`RobotController`)
- **功能**: 整合Brain和Body，实现完整的感知-决策-执行循环
- **核心职责**:
  1. 接收用户输入 (控制台或ROS)
  2. 调用LLM进行决策推理
  3. 执行机器人动作
  4. 管理对话历史
  5. 统计交互信息
- **关键方法**:
  - `process_input()`: 处理用户输入的完整流程
  - `_execute_action()`: 执行具体动作
  - `interactive_mode()`: 交互模式(命令行对话)
  - `run_test_scenario()`: 运行测试场景

### 4. Bridge层 (桥接)

**位置**: `bridge/`

#### 4.1 ROS语音桥接器 (`ros_voice_bridge.py`)
- **功能**: 作为CADE系统的ROS入口，负责语音交互的完整流程
- **架构演变**:
  - Pull Mode (旧): stdin → RobotController → print
  - Push Mode (新): /asr → RosVoiceBridge → RobotController → /tts
- **核心职责**:
  1. **感知**: 监听`/asr`话题，拦截无效输入(回声抑制)
  2. **认知**: 调用RobotController进行LLM推理
  3. **执行**: 发布回复到`/tts`话题
- **关键特性**:
  - 状态锁(`_state_lock`): 用于线程安全
  - 异步处理: 在新线程中处理输入，避免阻塞ROS回调
  - 回声抑制: 检查`robot.is_busy()`，忽略机器人忙碌时的输入

### 5. ASR/TTS层 (语音处理)

**位置**: `src/asr_tts/scripts/`

#### 5.1 ASR节点 (`asr_node.py`)
- **功能**: 语音识别节点，使用Sherpa-ONNX进行实时语音识别
- **技术栈**:
  - **语音活动检测**: Silero VAD模型
  - **语音识别**: SenseVoice多语言模型(支持中、英、日、韩、粤语)
  - **音频处理**: sounddevice用于音频输入，numpy用于数据处理
- **工作流程**:
  1. 初始化音频设备，锁定指定麦克风
  2. 创建不间断流式录音
  3. VAD检测语音活动
  4. ASR识别语音内容
  5. 发布识别结果到`/asr`话题
- **关键特性**:
  - 生产者-消费者模式: 音频队列避免阻塞
  - 设备采样率重采样: 支持不同采样率设备
  - 音频保存: 可选保存录音文件

#### 5.2 TTS节点 (`tts_node.py`)
- **功能**: 语音合成节点，使用Sherpa-ONNX进行文本转语音
- **技术栈**:
  - **语音合成**: VITS中文模型(vits-zh-aishell3)
  - **音频播放**: sounddevice或系统播放器(paplay)
- **工作流程**:
  1. 初始化TTS引擎
  2. 订阅`/tts`话题
  3. 接收文本并生成语音
  4. 保存音频文件并播放
- **关键特性**:
  - 设备锁定: 确定性匹配输出设备
  - 多架构支持: VITS、Kitten等模型架构
  - 播放策略: 优先使用系统播放器，回退到sounddevice

### 6. 配置层

**位置**: `config.py`

#### 6.1 配置抽象层 (`Config`类)
- **功能**: 支持云端(CLOUD)和本地(LOCAL)模式无缝切换
- **配置层次**:
  1. **环境变量** (最安全): DEEPSEEK_API_KEY, MODE等
  2. **本地配置文件** (推荐): `config_local.py`
  3. **默认配置**: `config.py`中的默认值
- **运行模式**:
  - `RunMode.CLOUD`: 云端API模式 (当前PC开发)
  - `RunMode.LOCAL`: 本地Ollama模式 (未来Orin部署)
- **关键配置项**:
  - LLM参数: base_url, api_key, model, temperature, max_tokens, timeout
  - 机器人配置: ROBOT_NAME, ENABLE_MOCK
  - 日志配置: LOG_LEVEL, LOG_FILE

### 7. 入口点

#### 7.1 控制台入口 (`main.py`)
- **功能**: 交互式控制台模式的入口点
- **特性**: 支持对话、测试场景、统计信息显示

#### 7.2 ROS语音入口 (`main_ros.py`)
- **功能**: ROS语音模式的入口点
- **特性**:
  - 打印欢迎信息和测试案例
  - 支持多种提示词模式(default, simple, compact, debug)
  - 可配置环境上下文
  - 启动RosVoiceBridge并进入ROS主循环

## 数据流与交互流程

### 控制台模式数据流
```
用户输入 → main.py → RobotController → LLMClient → Brain决策 → 
Robot执行 → 控制台输出
```

### ROS语音模式数据流
```
麦克风输入 → asr_node.py → /asr话题 → RosVoiceBridge → RobotController → 
LLMClient → Brain决策 → Robot执行 → /tts话题 → tts_node.py → 扬声器输出
```

### 单次交互详细流程
1. **语音输入**: 用户说话，ASR节点识别并发布到`/asr`
2. **状态检查**: RosVoiceBridge检查机器人状态，如果忙碌则忽略
3. **LLM推理**: 调用RobotController.process_input()，LLM生成决策
4. **动作执行**: 执行决策中的动作(navigate/search/pick/place/speak/wait)
5. **语音输出**: 提取reply内容，发布到`/tts`话题
6. **状态更新**: 更新机器人状态，准备下一轮交互

## 关键设计决策

### 1. 模块化架构
- **决策**: 将系统划分为Brain、Body、Bridge、Controller层
- **理由**: 关注点分离，便于独立开发、测试和替换组件
- **优势**: 高内聚低耦合，易于扩展和维护

### 2. 抽象接口设计
- **决策**: 定义`RobotInterface`抽象类
- **理由**: 支持Mock和Real机器人的无缝切换
- **优势**: 便于开发测试(无硬件)和实际部署(有硬件)

### 3. 双模式运行
- **决策**: 支持CLOUD和LOCAL两种LLM运行模式
- **理由**: 适应不同部署场景(开发/测试/生产)
- **优势**: 开发阶段使用云端API快速迭代，部署阶段使用本地模型保证隐私

### 4. 状态管理与回声抑制
- **决策**: 实现`RobotState`状态机和`is_busy()`方法
- **理由**: 防止语音交互中的回声和重复触发
- **优势**: 提升交互自然度，避免机器人自言自语

### 5. 严格的JSON输出格式
- **决策**: 使用Pydantic强制LLM输出结构化JSON
- **理由**: 确保LLM输出可解析、可验证
- **优势**: 提高系统鲁棒性，减少解析错误

## 扩展点与定制化

### 1. 添加新动作类型
1. 在`brain/schemas.py`中定义新动作类
2. 在`brain/prompts.py`中更新系统提示词
3. 在`body/robot.py`中实现动作逻辑
4. 在`robot_controller.py`的`_execute_action()`中添加分支

### 2. 支持新LLM提供商
1. 在`config.py`中添加新提供商配置
2. 在`brain/llm_client.py`中扩展API调用逻辑
3. 更新`Config.get_llm_config()`方法

### 3. 添加新输入/输出模态
1. 在`bridge/`目录中创建新的桥接模块
2. 更新`robot_controller.py`支持新接口
3. 创建对应的入口点文件

### 4. 集成新机器人平台
1. 实现`RobotInterface`接口的新类
2. 在`body/`目录中添加实现
3. 通过配置切换机器人实例

## 部署架构

### 开发环境 (PC)
- **LLM**: 云端模式 (DeepSeek API)
- **语音**: 本地麦克风和扬声器
- **机器人**: Mock模式 (模拟执行)

### 生产环境 (机器人硬件)
- **LLM**: 本地模式 (Ollama + Qwen3)
- **语音**: 机器人内置麦克风和扬声器
- **机器人**: Real模式 (实际硬件控制)
- **中间件**: ROS (机器人操作系统)

### 硬件要求
- **开发环境**: Python 3.8+, 麦克风, 扬声器, 网络连接
- **生产环境**: NVIDIA Jetson Orin (推荐), ROS支持, 机器人硬件

## 故障排除与调试

### 常见问题
1. **LLM连接失败**: 检查API密钥和网络连接
2. **语音识别不准**: 调整麦克风位置，检查环境噪音
3. **回声问题**: 确认回声抑制逻辑正常工作
4. **动作执行失败**: 检查机器人状态和物体数据库

### 调试模式
- 启用debug提示词模式: `--mode debug`
- 显示详细思考过程: 默认启用，可用`--no-thought`禁用
- 查看原始LLM输出: 在`llm_client.py`中打印

## 总结

CADE系统通过模块化、可扩展的架构设计，实现了具身AI机器人的核心功能。系统严格分离了智能推理(Brain)、物理执行(Body)、系统集成(Bridge)和业务协调(Controller)，使得每个组件可以独立发展、测试和替换。双模式运行(云端/本地)和抽象接口设计进一步提高了系统的适应性和可部署性。

该架构既满足了当前家庭服务机器人的需求，也为未来的功能扩展和技术演进奠定了坚实基础。
